# 边缘大脑系统 (Edge Brain Open System  简写 edgeOS) 规划方案

## 1. 项目概述

### 1.1 项目背景
针对现有的边缘采集系统 EdgeX 存在以下问题：
- 单点故障风险高，缺乏有效的冗余机制
- 设备发现和双向数据通信能力不足
- 缺乏统一的调度和协调机制
- 影子设备消息共享机制不完善

### 1.2 项目目标
设计并实现一个功能完善的边缘大脑系统 edgeOS ，该系统应包含性能强劲的主控节点，负责执行各类调度采集任务、边缘群控算法及节能优化算法。实现N+2冗余架构模式，其中N代表多个分布式边缘采集网关程序EdgeX，2代表本地主控大脑的双机热备机制（即蜂群模式），包含一个主母皇节点 edgeOS 与一个备用母皇节点 edgeOS 。该架构需能够协调N个边缘采集网关程序 EdgeX，实现 EdgeX 设备自动发现、自动注册EdgeX , 支持EdgeX 的节点监控, 自动同步 EdgeX 里面的影子设备 与EdgeX双向数据通信及其通过影子设备下发来控制真实设备核心功能。

### 1.3 设计原则
- **高可用性**：N+2冗余架构，确保系统持续运行
- **可扩展性**：支持横向扩展，适应不同规模的部署
- **实时性**：毫秒级响应，满足工业现场实时需求
- **一致性**：分布式环境下数据一致性保证
- **自愈性**：故障自动检测和恢复

### 1.4 应用场景

- **工业物联网**：工厂设备的实时监控、数据采集和远程控制。
- **智能城市**：交通、安防、环境监测等设备的集中管理。
- **智慧园区**：园区设备的智能化管理和能源优化。
- **边缘云**：边缘节点的资源管理和任务调度。

### 2.2 N+2 冗余架构

#### 2.2.1 架构组成

|------|------|------|------|
| 边缘采集网关 (EdgeX) | N | 分布式部署的采集程序 | 负责设备连接、数据采集、协议转换 |

### 4.3 数据流向

1. **设备 → EdgeX**：设备数据通过 EdgeX 边缘采集网关采集。
2. **EdgeX → edgeOS**：EdgeX 将采集的数据同步到 edgeOS 主母皇节点。
3. **edgeOS → EdgeX**：edgeOS 将设备配置、控制命令等同步到 EdgeX。
4. **EdgeX → 设备**：EdgeX 将控制命令下发到设备。

### 4.4 数据类型

- **遥测数据**：设备的实时采集数据，如温度、湿度、压力等。
- **控制命令**：对设备的控制指令，如开关、调节等。
- **配置数据**：设备的配置信息，如采集频率、阈值等。
- **状态数据**：设备和节点的状态信息，如在线状态、运行状态等。
- **告警数据**：设备和系统的告警信息，如故障、超限等。

### 7.3 EdgeX 影子设备与 EdgeOS 实时同步机制

#### 7.3.1 EdgeOS 角色定位
- **EdgeOS 不直接参与数据采集工作**，仅负责接收并实时同步来自 EdgeX 的影子设备数据
- **EdgeOS 作为影子设备数据的汇聚和管理中心**，负责影子设备的注册、状态同步和消息广播
- **EdgeOS 提供统一的接口**，供外部系统（如北向适配器）访问影子设备数据

#### 7.3.2 同步实现步骤
1. **初始化同步**：
   - EdgeX 与 EdgeOS 建立连接，进行握手和身份认证
   - EdgeX 向 EdgeOS 同步所有影子设备的初始状态（全量同步）

2. **实时同步**：
   - EdgeX 采集设备数据后，更新本地影子设备状态
   - EdgeX 通过消息总线向 EdgeOS 发送影子设备状态变更消息（增量同步）
   - EdgeOS 接收并处理影子设备状态变更消息，更新本地注册表
   - EdgeOS 向备用母皇节点同步影子设备状态变更（跨节点同步）

3. **定时对账**：
   - EdgeOS 定期与 EdgeX 进行影子设备状态对账
   - 检测并解决可能的数据一致性问题
   - 确保 EdgeOS 和 EdgeX 之间的数据一致性

#### 7.3.3 数据传输协议
- **消息总线**：采用可靠的消息队列，如 Kafka、RabbitMQ 等
- **通信协议**：支持 MQTT、AMQP、HTTP 等多种通信协议
- **数据格式**：采用 JSON 或 Protocol Buffers 格式，确保数据传输的效率和可靠性
- **QoS 保证**：支持不同级别的服务质量，确保消息传输的可靠性

#### 7.3.4 同步频率要求
- **实时同步**：设备状态变更时立即同步，延迟不超过 100ms
- **定时对账**：每 5 分钟进行一次全量对账，确保数据一致性
- **心跳检测**：每 10 秒进行一次心跳检测，确保连接的可靠性

#### 7.3.5 异常处理机制
- **连接异常**：当 EdgeX 与 EdgeOS 之间的连接中断时，EdgeX 会缓存影子设备状态变更，待连接恢复后进行批量同步
- **同步失败**：当同步消息发送失败时，会进行重试，重试次数可配置（默认 3 次）
- **数据冲突**：当 EdgeOS 和 EdgeX 之间出现数据冲突时，根据时间戳和版本号进行冲突解决，默认以 EdgeX 的数据为准
- **节点故障**：当主母皇节点故障时，备用母皇节点会接管同步工作，确保同步过程不中断

#### 7.3.6 验证方法
- **功能验证**：验证 EdgeX 影子设备状态变更是否能正确同步到 EdgeOS
- **性能验证**：验证同步延迟是否满足要求，同步吞吐量是否能满足系统需求
- **可靠性验证**：验证在网络中断、节点故障等异常情况下，同步机制是否能正常工作
- **一致性验证**：验证 EdgeOS 和 EdgeX 之间的影子设备数据是否一致

### 7.4 数据同步机制

#### 7.4.1 同步对象

- **节点信息**：包括主母皇节点、备用母皇节点和 EdgeX 边缘采集网关的基本信息、状态、角色等。
- **设备信息**：包括设备的基本信息、状态、配置等。
- **点位信息**：包括设备的点位定义、采集配置、阈值等。
- **控制命令**：包括对设备的控制指令、执行状态等。
- **告警信息**：包括设备和系统的告警信息。

#### 7.4.2 同步策略

- **全量同步**：系统初始化或故障恢复时，进行全量数据同步。
- **增量同步**：正常运行时，仅同步发生变化的数据。
- **事件驱动**：数据发生变化时，触发同步操作。
- **定期同步**：定期进行全量或增量同步，确保数据一致性。

#### 7.4.3 冲突处理

- **冲突检测**：检测同步过程中的数据冲突。
- **冲突解决**：根据预设的规则，解决数据冲突。
- **冲突记录**：记录冲突的详细信息，便于后续分析。
